Description: 3x daily search for new OpenClaw/Hermes/Claude agent use cases across X, Reddit, HN, GitHub, ProductHunt, IndieHackers — write to Firestore

Prompt:
You are running the OpenClaw use-case ingest job on a local agent (OpenClaw / Hermes). Your goal is to find 5–15 NEW, non-duplicate use cases of OpenClaw / Hermes / Claude agents being used in real workflows, with special priority on (a) revenue-generating deployments and (b) multi-step / iterative / long-running agent workflows that go beyond a single one-shot prompt.

Freshness matters. Prefer genuinely recent findings first, and only fall back to older source material when you cannot find enough strong recent items.

The working directory for this job is defined by the environment variable $OPENCLAW_SCHEDULED_DIR. All file reads and writes below use that path as the base.



**Product terms to search for:** "openclaw" OR "hermes" (the AI agent) OR "claude cowork" OR "claude agent" — use all of these interchangeably across platforms.



## Step 1 — Load existing records to avoid dupes



Read `$OPENCLAW_SCHEDULED_DIR/pending_records_*.json` (most recent file) and `$OPENCLAW_SCHEDULED_DIR/seen_ids.txt` if it exists. Extract all source URLs and IDs to skip later.



## Step 2 — Run all searches

Freshness policy for search and selection:
- Strongly prefer items whose original post/article date is within the last 7 days relative to the run date.
- If you cannot find enough high-quality items from the last 7 days, expand to the last 30 days.
- Review candidates newest-first whenever source dates can be determined.
- For a normal batch of roughly 10 selected items, require at least 2 items from the last 7 days whenever the search results make that possible; aim for 2–3.
- If you do not yet have 2 last-7-day items, keep searching before finalizing unless you have genuinely exhausted the available recent sources.
- Only use older items when they are exceptionally strong and you still do not have enough recent qualifying results.
- Any selected item older than 30 days must be treated as a fallback item and explicitly flagged as such in the final report.
- When an older item is included, treat that as an exception and mention in the final report that it was chosen as a fallback due to insufficient recent supply.
- If multiple candidates are similar in quality, always choose the more recent one.
- Do not spend scarce slots on old posts just because they are new to this database; recency should win unless the older item is materially better.
- Product Hunt should be treated as a supplementary source for freshness-sensitive runs, not a primary source for filling the recent-item quota.



Use WebSearch for general and authenticated-wall sources (X, Reddit web). Use WebFetch for public JSON APIs (HN Algolia, Reddit JSON API, IndieHackers). Wait 3–5 seconds between requests to avoid rate limiting.

## Step 3 — Filter and score



For each result, ask: does this describe a **real, specific deployment** of OpenClaw / Hermes / a Claude agent **running autonomously** in an actual workflow? The agent must be the *actor* — doing things on its own, not merely assisting a human who is doing the work. Also ask: is it recent enough to deserve one of today’s slots?

**The core litmus test:** After the thing described is done, is an OpenClaw / Hermes / Cowork / Claude agent *still* running and doing things autonomously? If the answer is no — if the "agent" was just a one-off assistant the human directed — discard the result.

Discard:

- Generic tutorials, how-to guides, Anthropic marketing

- Speculation or "I’m thinking of building..." posts

- Results matching any URL/ID from Step 1 (dupes)

- Near-duplicates of each other (same use case, different post)

- Older candidates that are weaker than available newer candidates

- **"Built with Claude/AI" stories** — posts where a person used Claude Code, Claude chat, or any LLM *as a coding or writing assistant* to build a product, but the finished product itself does not contain or run an agent. Example to reject: "I used Claude Code to help me write the Rust core of my macOS app." The finished app is not an agent — discard. The test: does the *deployed product or workflow* contain an agent that keeps running after the build phase?

- **One-shot prompting showcases** — a single human prompt produces a single output, even if the output is impressive. Must involve scheduling, memory, sub-agents, iteration, or a multi-step autonomous pipeline to qualify.

- **Pure product-announcement posts** with no workflow evidence — a launch tweet or ProductHunt card that describes what a product *will* do, with no sign of real users or a running deployment.

- **Comparison / overview articles** that survey the agent landscape but do not themselves document any specific deployment (e.g. "OpenClaw vs Claude Cowork" roundups, "Top 10 use cases" listicles).



Tag each qualifying result:

- `category`: Marketing / Productivity / Engineering / Research / Finance / Creative / Other

- `revenue_signal`: true/false — mentions $, ARR, customers, subscribers, income?

- `deep_reasoning`: true/false — involves multi-step iteration, long-running agents, multi-agent coordination, or synthesis across many sources — NOT a single one-shot big-model prompt?

- `uncertainty`: low / medium / high

- `novelty`: low / medium / high



## Step 4 — Build records



For each qualifying use case, produce a JSON object with EXACTLY these keys (matching what ingest.py expects):

Date selection rules:
- `tweetDate` must be the original source/post/article date when it can be determined reliably.
- `searchDate` must be the current run date.
- If the source date is unclear, do not invent a recent `tweetDate`; leave it empty rather than fabricating freshness.



```json

{

  "category": "Marketing|Productivity|Engineering|Research|Finance|Creative|Other",

  "sourceUser": "name of author or publisher",

  "description": "2–3 sentence factual description of exactly what the agent is doing, what the outcome is, and any quantified results (time saved, revenue, scale)",

  "refUrls": "https://source-url-here",

  "tweetDate": "YYYY-MM-DD",

  "searchDate": "YYYY-MM-DD",

  "notes": "id=<sha256_of_refUrls_first_12_hex_chars>; revenue_signal=True|False; deep_reasoning=True|False",

  "uncertainty": "low|medium|high",

  "novelty": "low|medium|high"

}

```



Key mapping rules:

- `sourceUser` = the author or publisher (NOT "author")

- `description` = the 2–3 sentence summary (NOT "summary")

- `refUrls` = the source URL string (NOT "source_url")

- `tweetDate` = the date of the original post/article (NOT "date")

- `searchDate` = today's date (the date this ingest job ran)

- `notes` = pack the sha256 id, revenue_signal, and deep_reasoning as a single string: `"id=<hash>; revenue_signal=True; deep_reasoning=False"`

- Do NOT include top-level keys named `id`, `author`, `date`, `source_url`, `revenue_signal`, `deep_reasoning`, or `summary` — these will be rejected by ingest.py



## Step 5 — Write output



Save the array of records to `$OPENCLAW_SCHEDULED_DIR/pending_records_YYYY-MM-DD_HH.json` (include hour to avoid overwriting sibling runs).



Also update `$OPENCLAW_SCHEDULED_DIR/seen_ids.txt` by appending any new IDs (one per line).



Then run:

```

cd $OPENCLAW_SCHEDULED_DIR && cat pending_records_YYYY-MM-DD_HH.json | python3 ingest.py

```



If egress to oauth2.googleapis.com is blocked (proxy error), leave the pending file and note it clearly so the user can run ingest.py locally.

- how many selected records were from the last 30 days
- how many were older fallback items

- whether Product Hunt contributed any selected items and, if so, whether they were used only as supplementary/fallback inputs
