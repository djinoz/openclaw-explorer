Description: 3x daily search for new OpenClaw/Hermes/Claude agent use cases across X, Reddit, HN, GitHub, ProductHunt, IndieHackers — write to Firestore

Prompt:
You are running the OpenClaw use-case ingest job on a local agent (OpenClaw / Hermes). Your goal is to find 5–15 NEW, non-duplicate use cases of OpenClaw / Hermes / Claude agents being used in real workflows, with special priority on (a) revenue-generating deployments and (b) multi-step / iterative / long-running agent workflows that go beyond a single one-shot prompt.

The working directory for this job is defined by the environment variable $OPENCLAW_SCHEDULED_DIR. All file reads and writes below use that path as the base.



**Product terms to search for:** "openclaw" OR "hermes" (the AI agent) OR "claude cowork" OR "claude agent" — use all of these interchangeably across platforms.



## Step 1 — Load existing records to avoid dupes



Read `$OPENCLAW_SCHEDULED_DIR/pending_records_*.json` (most recent file) and `$OPENCLAW_SCHEDULED_DIR/seen_ids.txt` if it exists. Extract all source URLs and IDs to skip later.

## Step 3 — Filter and score



For each result, ask: does this describe a **real, specific deployment** of OpenClaw / Hermes / a Claude agent in an actual workflow? Discard:

- Generic tutorials, how-to guides, Anthropic marketing

- Speculation or "I'm thinking of building..." posts

- Results matching any URL/ID from Step 1 (dupes)

- Near-duplicates of each other (same use case, different post)



Tag each qualifying result:

- `category`: Marketing / Productivity / Engineering / Research / Finance / Creative / Other

- `revenue_signal`: true/false — mentions $, ARR, customers, subscribers, income?

- `deep_reasoning`: true/false — involves multi-step iteration, long-running agents, multi-agent coordination, or synthesis across many sources — NOT a single one-shot big-model prompt?

- `uncertainty`: low / medium / high

- `novelty`: low / medium / high



## Step 4 — Build records



For each qualifying use case, produce a JSON object with EXACTLY these keys (matching what ingest.py expects):



```json

{

  "category": "Marketing|Productivity|Engineering|Research|Finance|Creative|Other",

  "sourceUser": "name of author or publisher",

  "description": "2–3 sentence factual description of exactly what the agent is doing, what the outcome is, and any quantified results (time saved, revenue, scale)",

  "refUrls": "https://source-url-here",

  "tweetDate": "YYYY-MM-DD",

  "searchDate": "YYYY-MM-DD",

  "notes": "id=<sha256_of_refUrls_first_12_hex_chars>; revenue_signal=True|False; deep_reasoning=True|False",

  "uncertainty": "low|medium|high",

  "novelty": "low|medium|high"

}

```



Key mapping rules:

- `sourceUser` = the author or publisher (NOT "author")

- `description` = the 2–3 sentence summary (NOT "summary")

- `refUrls` = the source URL string (NOT "source_url")

- `tweetDate` = the date of the original post/article (NOT "date")

- `searchDate` = today's date (the date this ingest job ran)

- `notes` = pack the sha256 id, revenue_signal, and deep_reasoning as a single string: `"id=<hash>; revenue_signal=True; deep_reasoning=False"`

- Do NOT include top-level keys named `id`, `author`, `date`, `source_url`, `revenue_signal`, `deep_reasoning`, or `summary` — these will be rejected by ingest.py



## Step 5 — Write output



Save the array of records to `$OPENCLAW_SCHEDULED_DIR/pending_records_YYYY-MM-DD_HH.json` (include hour to avoid overwriting sibling runs).



Also update `$OPENCLAW_SCHEDULED_DIR/seen_ids.txt` by appending any new IDs (one per line).



Then run:

```

cd $OPENCLAW_SCHEDULED_DIR && cat pending_records_YYYY-MM-DD_HH.json | python3 ingest.py

```



If egress to oauth2.googleapis.com is blocked (proxy error), leave the pending file and note it clearly so the user can run ingest.py locally.



## Step 6 — Report



Print a markdown summary table of all new records, tagged by revenue_signal and deep_reasoning. State: searches run, raw candidates reviewed, dupes skipped, new records written, Firestore status.
